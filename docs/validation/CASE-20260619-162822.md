## Test objective

Confirm that the v2 entry point can load the module, discover and execute all initial plugins, export evidence, normalize findings, and render the enterprise HTML report on a Windows endpoint.

## Successful outputs

The assessment generated a complete case folder containing:

- `case-metadata.json`
- `summary.json`
- `plugin-results.json`
- `findings.csv`
- `enterprise-case-report.html`

The HTML report opened successfully in the default browser.

## Initial findings summary

The first run generated findings from all four initial plugins:

- Windows Update errors and pending reboot state
- An APIPA address observed by the network plugin
- Repeated Windows event groups
- Update service state context

Representative event groups included:

- DistributedCOM 10016
- PowerShell 4100 and 4104
- Windows Update Client 20
- .NET Runtime 1022
- VBScript deprecation telemetry 4096
- NDIS 10317
- Wireless adapter telemetry 6062
- Time Service 134
- Kernel Processor Power 37
- Hyper-V virtual switch 22

## Review findings

The run proved that the plugin engine and report workflow functioned, but it also exposed tuning requirements:

1. APIPA detection reviewed all interfaces and could promote an address from an inactive or virtual interface.
2. Manual or trigger-start Windows Update services could be reported simply because they were stopped.
3. Event correlation promoted event groups primarily by count and level, which over-scored common or expected telemetry.
4. PowerShell operational events could include telemetry generated by the assessment itself.

## Changes made after validation

### Network Diagnostics 2.0.1

- APIPA findings now require an active physical interface or a default-route interface.
- APIPA addresses on non-routing virtual interfaces remain available as evidence without automatically becoming high-severity findings.
- Public DNS checks now focus on active or routing-relevant interfaces.

### Windows Update 2.0.1

- Stopped manual or trigger-start services are no longer automatically reported as faults.
- Disabled services remain high-severity findings.
- Stopped automatic services remain actionable.
- Update error severity now considers recency and repetition.

### Event Correlation 2.0.1

- Added event-specific tuning profiles.
- DistributedCOM 10016 is retained as evidence unless it directly matches the symptom.
- PowerShell 4100 and 4104 are retained as evidence instead of promoted by repetition alone.
- VBScript deprecation telemetry is retained as compatibility evidence.
- Actionable events receive conservative event-specific severity and recommendations.
- Generic warning events are no longer promoted to High solely because of count.

## Validation conclusion

The initial v2 architecture passed its first Windows runtime test. The test also demonstrated the value of real execution evidence: the framework produced usable output and immediately identified areas where rule quality needed refinement.
